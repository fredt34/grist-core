# Chat 1 ÔÇö Brainstorm: per-document CSS injection

## Context

Goal: apply per-document CSS rules to a Grist document, stored as a document setting (alongside timezone, locale, currencyÔÇĞ), and injected client-side when the document loads.

---

## Analysis & Critique of the original `context.md` plan

### Ô£à What's Right

1. **`documentSettingsJson` is the correct storage mechanism.** The existing pattern (`locale`, `currency`, `attachmentStoreId`) is exactly the right place to add a `customCSS` field. It's stored in `_grist_DocInfo`, persisted with the document, and already has a clean observable/save pattern in `DocInfoRec`.

2. **`DocSettingsPage` is the right UI location.** It already has the `SectionCard`/`SectionItem` pattern, owner-only guards, and the save-to-docInfo plumbing.

3. **"Keep it simple and light"** ÔÇö good instinct. This is a fork-specific feature, not upstream material.

---

### ÔØî Critical Problems

#### 1. The injection mechanism described in context.md is wrong

The plan said: *"`ICreate.ts` should read the configuration option `documentCSS`... and inject it in `/static/app.html` by extending `getExtraHeadHtml`"*

This is architecturally impossible for per-document CSS:

- `getExtraHeadHtml()` is called once per HTTP request in `sendAppPage.ts`, producing **static HTML** that is the same for all documents.
- `documentCSS` is a **per-document setting** stored inside the document's SQLite data. The server doesn't know which document is being opened when it renders `app.html` ÔÇö the document is loaded later, client-side, via WebSocket.
- `ICreate.ts` has no access to document data. It's a server-level factory.

**The CSS cannot be injected server-side via `getExtraHeadHtml` for per-document CSS.** That mechanism only works for server-wide CSS (which already exists via `APP_STATIC_INCLUDE_CUSTOM_CSS`).

#### 2. The right injection point is the client, not the server

The CSS is available client-side once the document loads ÔÇö `docInfo.documentSettingsJson` is a reactive observable. The correct approach is to **inject a `<style>` tag into the DOM from the client** when the document opens, and remove/update it when the document closes or the setting changes.

#### 3. The "Apply to Owners only / Apply to all users" debug idea is confused

If CSS is injected client-side (which it must be), visibility control is trivially implementable by checking the user's role before injecting. The more useful debug toggle would be a **live preview** in the settings page before committing.

#### 4. Security

CSS can exfiltrate data via `url()` requests combined with attribute selectors. Since this is owner-only, the risk is self-inflicted. Reasonable stance: only owners can set it.

#### 5. Parameter name

`documentCSS` ÔåÆ renamed to **`customCSS`** for consistency with `APP_STATIC_INCLUDE_CUSTOM_CSS`.

---

## Agreed Architecture

### Storage
Add `customCSS?: string` to `DocumentSettings` interface in `app/common/DocumentSettings.ts`.  
Expose via `documentSettingsJson.prop("customCSS")` in `DocInfoRec`.

### UI
New `SectionCard` in `DocSettingsPage` with a `<textarea>`, owner-only, with Save + live Preview.

### Injection (client-side)
In `GristDocImpl` constructor, subscribe to the `customCSS` observable and manage a `<style id="grist-custom-doc-css">` tag in `document.head`. Remove it on document dispose.

### 4. `app/client/components/GristDoc.ts`
In `GristDocImpl` constructor, after `this.docInfo = this.docModel.docInfoRow;`:

```typescript
// Inject per-document custom CSS into document.head, reactively.
this.autoDispose(
  fromKo(this.docInfo.customCSS).addListener((css) => {
    let styleEl = document.getElementById('grist-custom-doc-css') as HTMLStyleElement | null;
    if (!styleEl) {
      styleEl = document.createElement('style');
      styleEl.id = 'grist-custom-doc-css';
      document.head.appendChild(styleEl);
    }
    styleEl.textContent = css || '';
  })
);

this.onDispose(() => {
  const styleEl = document.getElementById('grist-custom-doc-css');
  if (styleEl) { styleEl.remove(); }
});
```

### Testing via browser console

```javascript
// Write a customCSS value ÔÇö use the actual current settings to avoid overwriting other fields
gristDocPageModel.gristDoc.get().docInfo.documentSettings.saveOnly(
  JSON.stringify({
    ...JSON.parse(gristDocPageModel.gristDoc.get().docInfo.documentSettings.peek() || '{}'),
    customCSS: '.column_name { color: red !important; font-weight: bold !important; }'
  })
);
```

> **Note:** `body { background: ... }` doesn't work well because Grist sets backgrounds on inner elements.
> Target specific Grist CSS classes instead (e.g. `.column_name`, `.gridview_data_row_num`).
